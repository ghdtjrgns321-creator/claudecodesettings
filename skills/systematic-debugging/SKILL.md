# Systematic Debugging — 원인 없이 고치지 않는다

> superpowers 6.1.1에서 발췌·재작성 (의례 제거)

무작위 수정은 시간을 태우고 새 버그를 만든다. 임시방편은 진짜 문제를 가린다.

**철칙: 근본 원인을 찾기 전에 수정하지 않는다.** 증상 수정은 실패다.

## 4단계

### 1. 원인 조사 (수정 시도 전)

- **에러 메시지를 끝까지 읽는다.** 스택트레이스 전체, 줄 번호, 파일 경로, 에러 코드. 답이 거기 있는 경우가 많다.
- **재현한다.** 확실히 띄울 수 있나? 매번 나나? 재현이 안 되면 데이터를 더 모은다 — 추측하지 않는다.
- **최근 변경을 본다.** git diff, 최근 커밋, 새 의존성, 설정 변경, 환경 차이.
- **다층 시스템이면 계층마다 계측한다.** 컴포넌트 경계마다 들어가는 값·나오는 값을 찍어서 **어디서 깨지는지** 먼저 특정하고, 그 컴포넌트를 판다. 추측으로 층을 건너뛰지 않는다.
- **데이터 흐름을 거슬러 올라간다.** 나쁜 값이 어디서 태어났나 → 누가 그 값으로 호출했나 → 발원지까지. **발원지를 고치지 증상을 고치지 않는다.**

### 2. 패턴 분석

- 같은 코드베이스에서 **작동하는 비슷한 코드**를 찾는다.
- 참조 구현이 있으면 **전체를 읽는다.** 훑지 말 것 — 부분 이해는 버그를 보장한다.
- 작동하는 것과 깨진 것의 **차이를 전부 나열**한다. "이건 상관없겠지"를 하지 않는다.

### 3. 가설과 시험

- **가설 하나를 명시한다.** "X가 원인이다, 왜냐면 Y". 모호하지 않게.
- **최소로 시험한다.** 한 번에 한 변수. 여러 개를 동시에 고치지 않는다.
- 안 되면 **새 가설을 세운다.** 실패한 수정 위에 수정을 얹지 않는다.
- **모르면 모른다고 말한다.** 아는 척하지 않는다.

### 4. 구현

- **실패하는 테스트/재현 케이스를 먼저** 만든다. 프레임워크가 없으면 일회용 스크립트라도.
- **수정은 하나씩.** "온 김에" 개선·번들 리팩터링 금지.
- **검증한다.** 통과하나? 다른 게 깨지지 않았나? 실제로 해결됐나?

## 3회 규칙 — 수정이 3번 실패하면 아키텍처를 의심한다

수정 시도를 세라. **3번째부터는 4번째 수정을 시도하지 말고 멈춘다.**

아키텍처 문제의 징후:
- 고칠 때마다 **다른 곳에서** 새 문제가 튀어나온다
- 수정하려면 매번 "대규모 리팩터링"이 필요하다
- 각 수정이 새 증상을 만든다

이건 가설이 틀린 게 아니라 **구조가 틀린 것**이다. 사용자와 논의하고 나서 진행한다.

## 멈춰야 하는 신호 (내 생각 속에서)

- "일단 빨리 고치고 나중에 조사하자"
- "X를 바꿔보고 되나 보자"
- "여러 개 고쳐놓고 테스트 돌리자"
- "정확히는 모르겠는데 이러면 될 것 같다"
- "문제는 아마 X일 거다" (추적 없이)
- "한 번만 더 시도해보자" (이미 2번 실패)

**전부 1단계로 돌아가라는 뜻이다.**

## 사용자가 보내는 신호

- "그게 아니라니까" / "추측하지 말고" → 이해 없이 수정안을 내고 있다
- **"두더지잡기식 수정 아니야?" / "case by case로 묻어버리지 마" / "fitting 아니야?"** → 증상에 맞춰 분기를 늘리고 있다. 3회 규칙을 적용하라
- "처음부터 다시 생각해봐" → 근본을 의심하라는 뜻

## 흔한 합리화

| 변명                             | 현실                                                   |
| -------------------------------- | ------------------------------------------------------ |
| "간단한 이슈라 절차 필요 없다"   | 간단한 버그도 원인이 있다. 절차는 간단한 버그에 빠르다 |
| "급하니까 일단"                  | 추측-확인 반복이 체계적 조사보다 느리다                |
| "확인되면 테스트 나중에 쓰지"    | 검증 안 된 수정은 다시 돌아온다                        |
| "참조가 길어서 적당히 맞춰 쓰지" | 부분 이해 = 버그 보장                                  |
| "문제가 보인다, 바로 고치자"     | 증상을 본 것 ≠ 원인을 이해한 것                        |

## 원인이 진짜 없을 때

체계적 조사 결과 환경·타이밍·외부 요인이 확실하면: 조사한 내용을 기록하고, 적절한 처리(재시도·타임아웃·에러 메시지)를 넣고, 관측을 추가한다. **다만 "원인 없음"의 대부분은 조사가 덜 된 것이다.**
