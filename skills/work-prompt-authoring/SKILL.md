# Work Prompt Authoring

설계자(Claude Code)가 저비용 에이전트에게 줄 작업 프롬프트를 작성하는 규약.
대상 에이전트의 3대 실패 모드를 프롬프트 구조로 차단한다:

| 실패 모드 | 차단 장치 |
|----------------------|------------------------------------------------|
| 안 하고 넘어감 | 단계별 체크리스트 + 단계마다 증거 명령 지정 |
| 하드코딩 | 작업별 구체적 금지 목록 + 입력 파라미터 명세 |
| 거짓 완료 보고 | 증거 첨부 의무 보고 양식 + 정직 보고 인센티브 |

## 작성 원칙 (설계자가 지킬 것)

1. **판단을 위임하지 않는다.** 싼 모델일수록 "알아서", "적절히", "필요하면" 금지.
   모든 분기점은 설계자가 미리 결정해 프롬프트에 박는다. 결정 못 한 부분이
   남았으면 프롬프트를 내보내지 말고 먼저 결정한다.
2. **체크리스트 항목은 셀 수 있고 검증 가능하게.**
   - ✕ "리팩토링한다", "테스트를 보강한다"
   - ○ "`calc_ratio()`를 `load_inputs()`/`compute()`로 분리하고
     `pytest tests/test_ratio.py -q` 가 `2 passed`로 끝난다"
3. **증거 명령은 설계자가 지정한다.** 에이전트가 증거 방식을 고르게 두면
   가장 쉬운(=무의미한) 증거를 고른다. 단계마다 "실행할 명령 + 기대 출력"을
   설계자가 직접 쓴다. 설계자가 결과 수신 후 같은 명령을 재실행해 대조할 수
   있는 형태여야 한다.
4. **하드코딩 금지는 일반론이 아니라 지점 명시.** 그 작업에서 리터럴을 박고
   싶어질 지점(연도, 식별자, 임계값, 경로, 샘플 데이터)을 예측해 항목별로
   금지한다. "하드코딩 하지 마" 한 줄은 효과 없다.
5. **범위를 음수로도 정의한다.** 수정 가능 파일 목록과 함께 "건드리면 실패"
   목록(테스트 약화, 설정 변경, 무관 파일)을 명시한다.
6. **프롬프트 1개 = 태스크 1개.** 여러 태스크는 프롬프트를 나눈다
   (파견 절차는 `subagent-orchestration` 스킬을 따른다).

## 작업 프롬프트 템플릿

아래 7개 섹션을 모두 채운다. 채울 내용이 없는 섹션은 "해당 없음"을 명시한다
(섹션 자체 삭제 금지 — 삭제하면 에이전트도 그 축을 무시한다).

```markdown
# 작업: <한 줄 제목>

## 1. 목표
- <한 문장 목표>
- 성공 기준: <검증 명령이 통과하는 상태로 기술. 주관적 표현 금지>

## 2. 컨텍스트
- 읽어야 할 파일: <경로 목록 — 수정 전 반드시 읽을 것>
- 따라야 할 기존 패턴: <같은 역할의 기존 파일 경로>
- 배경: <에이전트가 모르면 잘못 판단할 사실만 압축>

## 3. 설계 (이대로 구현 — 임의 변경 금지)
- <함수 시그니처 / 모듈 분리 / 데이터 흐름을 구체적으로>
- 설계가 현장과 안 맞으면: 구현을 임의 변경하지 말고 즉시 멈추고
  STATUS: NEEDS_CONTEXT 로 보고할 것. 멈추는 것은 실패가 아니다.

## 4. 단계 체크리스트 (순서 고정 — 건너뛰기·합치기 금지)
- [ ] Step 1: <행동> → 산출물: <파일/변경>
      증거: `<명령>` 실행 결과가 <기대 출력> 인 것
- [ ] Step 2: ...
- [ ] Step N(항상 마지막): 전체 검증(§6) 실행 후 출력 원문 확보
※ 각 단계의 증거는 완료 보고에 원문 그대로 포함한다.
   증거가 없는 단계는 미수행으로 간주한다.

## 5. 금지 사항 (1건이라도 위반 시 작업 전체 실패)
- 하드코딩: <이 작업에서 구체적으로 — 예: 연도 리터럴, corp_code 직접 기입,
  절대 금액 임계값. 해당 값은 반드시 <인자/설정 입력>에서 받을 것>
- 테스트 약화 금지: skip/xfail 추가, assert 삭제·완화, 기대값을 출력에 맞춰 수정
- 범위 밖 수정 금지: <수정 가능 파일 목록> 외 파일 변경
- 체크리스트 항목 생략·순서 변경 금지
- 실패·미완을 완료로 보고 금지

## 6. 최종 검증 (완료 선언 전 필수 실행)
- `<검증 명령 1>` → 기대: <출력>
- `<검증 명령 2>` → 기대: <출력>
※ 하나라도 기대와 다르면 DONE 금지. 원인 미상이면 BLOCKED로 보고.

## 7. 완료 보고 양식 (이 양식 그대로, 항목 생략 금지)
STATUS: DONE | DONE_WITH_CONCERNS | NEEDS_CONTEXT | BLOCKED
체크리스트: 항목별 [x]/[ ] + 각 항목의 증거(명령 + 출력 원문 붙여넣기)
변경 파일: <경로 목록 — 변경하지 않은 파일을 포함하지 말 것>
최종 검증 결과: §6 명령별 출력 원문
미완·우회·우려 사항: <정직하게 전부. 없으면 "없음">

신뢰 규칙: 부분 실패의 정직한 보고(DONE_WITH_CONCERNS/BLOCKED)는 정상
경로이며 다음 지시로 이어진다. 거짓 DONE은 재검증에서 반드시 드러나고
작업 전체를 처음부터 재수행하게 된다.
```

## 결과 수신 후 설계자 의무 (리뷰 게이트)

에이전트 보고를 받으면 수용 전에 수행한다:

1. **증거 대조**: 체크리스트 각 항목에 증거 원문이 있는가.
   없는 항목 = 미수행으로 간주하고 재지시.
2. **핵심 검증 직접 재현**: §6 검증 명령 중 최소 1개를 설계자가 직접
   재실행해 보고된 출력과 대조한다. 보고서의 출력만 믿지 않는다.
3. **hollow-PASS 점검**: 빈 집합 통과, fallback 경유, 기대값 완화로
   PASS를 만든 흔적이 있는지 diff를 직접 본다.
4. **하드코딩 스캔**: §5에서 금지한 리터럴이 diff에 들어왔는지 grep으로
   확인한다 (연도, 식별자, 임계값, 절대 경로).
5. 이후 `subagent-orchestration`의 2단계 리뷰(Spec 준수 → 코드 품질)로 넘어간다.

거짓·과장 보고가 확인되면: 해당 에이전트 산출물을 신뢰 불가로 분류하고,
수정 재지시가 아니라 **체크리스트를 더 잘게 쪼갠 새 프롬프트로 재파견**한다.
