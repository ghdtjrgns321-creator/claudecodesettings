# Brainstorming — 코드 전에 가정을 좁힌다

목적: 잘못된 걸 빠르게 만드는 대신, 만들 것을 먼저 맞춘다.

**핵심 원칙: 상세도는 복잡도에 비례한다.** 간단한 건 몇 문장이면 설계 끝이다. 이 스킬은 의례가 아니다 — 단계를 채우는 게 목적이 아니라 모호함을 없애는 게 목적이다.

## 언제 쓰나

**쓴다:** 요구가 모호할 때 · 합리적 접근법이 2개 이상일 때 · 여러 파일·서브시스템에 걸칠 때 · 되돌리기 어려울 때

**안 쓴다:** 오타·명백한 버그 수정 · 이미 대화에서 결정된 절차의 반복 · 해가 하나뿐인 작업

## 절차

**1. 실물부터 본다.** 관련 파일·docs·최근 커밋. 상상으로 설계하지 않는다. 기존 패턴이 있으면 그걸 따른다.

**2. 범위를 먼저 잰다.** 독립 서브시스템이 여러 개면 즉시 분해를 제안한다. 큰 걸 통째로 설계하려 들지 않는다.

**3. 모르는 걸 묻는다 — 한 번에 하나씩.**
- 가능하면 객관식. 여러 질문으로 압도하지 않는다.
- 목적·제약·성공 기준에 집중.
- **답에 따라 결과가 달라지는 것만 묻는다.** 관행적 기본값이 있거나 코드에서 확인 가능한 건 그냥 확인하고 진행한다.

**4. 접근법 2~3개 + 트레이드오프.** 추천안을 먼저, 이유와 함께. 대화체로.

**5. 설계를 보인다.** 복잡도에 맞는 분량으로. 큰 설계면 섹션별로 확인받고, 작으면 한 번에.

**6. 승인 후 구현.** 설계에 동의를 받기 전에 코드를 쓰지 않는다.

## 설계 품질

- 작은 단위로 분해: 목적 하나, 명확한 인터페이스, 독립적으로 테스트 가능.
- 각 단위에 답할 수 있어야 한다 — 뭘 하나, 어떻게 쓰나, 뭘 의존하나.
- 내부를 안 읽고도 뭘 하는지 알 수 있나? 내부를 바꿔도 사용처가 안 깨지나? 아니면 경계가 잘못됐다.
- **YAGNI** — 투기성 기능은 설계에서 뺀다.
- 기존 코드에 문제가 있고 이 작업에 영향을 주면 그것도 설계에 포함. 무관한 리팩터링은 넣지 않는다.

## 산출

설계 문서는 **요청받았거나 세션을 넘겨야 할 때만** 쓴다. 이 대화에서 바로 구현할 거면 대화 자체가 설계다 — 문서를 만들고 커밋하는 의례를 붙이지 않는다.
