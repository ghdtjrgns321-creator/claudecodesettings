# FINAL-REPORT 장 스켈레톤

앞뒤 장은 고정, 중간 상세 장(3..K)만 프로젝트에 맞춰 적응 생성한다. 번호는 연속 부여.

```
FINAL-REPORT/
├── 0_README.md            # 표지: 배지 + 한줄요약 + 한눈 다이어그램 + 읽는 순서 + 핵심 수치
├── 1_OVERVIEW.md          # 문제 정의 → 왜 이 접근 → 아키텍처 원칙 → 기술 스택
├── 2_PIPELINE.md          # 전체 파이프라인 end-to-end (전체 설명 담당)
├── 3.._<PART>.md          # 부분별 상세 장 — 파이프라인 단계·데이터 여정 순서로 K-2개
├── K+1_DIFFERENTIATION.md # 기존 기술 대비 차별점·특장점 (근거 필수)
├── K+2_TEST-DECISIONS.md  # 검증(M/N) + 의사결정 ADR
├── K+3_JOURNEY.md         # 재구축 여정(아키텍처 전환·폐기·승계) — 과거형 전용
├── K+4_TROUBLESHOOTING.md # 운영 사고·디버깅 교훈(사고→근본원인→복구) — 과거형 전용
└── K+5_COVERAGE.md        # 전수 커버리지 부록 (census 대조 N/N)
```

## 장별 필수 요소

### 0_README.md (표지·목차)
1. **배지 줄** — shields.io static만: `![라벨](https://img.shields.io/badge/<label>-<value>-<color>?logo=<logo>&logoColor=white)`. 스택 버전·핵심 수치·검증 결과로 5~10개. 동적 배지(API 조회형) 금지.
2. **한 줄 요약** — 시스템이 무엇을 어떻게 다르게 하는지 1문장.
3. **시스템 한눈 ASCII 다이어그램** — 입력→종착 전체를 한 화면에.
4. **읽는 순서 표** — `# | 파일 | 무엇을 담나`.
5. **핵심 수치** — as-built 실측만, 각 수치에 상술 장 링크.

### 1_OVERVIEW.md
문제 정의(어떤 실패·위험을 막는 시스템인가) → 왜 이 접근인가(대안 대비) → 아키텍처 원칙 표 → 기술 스택 표(`레이어 | 기술 | 비고`).

### 2_PIPELINE.md (전체 설명 담당)
- 전체 흐름 대형 다이어그램 1개(ASCII 또는 Mermaid flowchart) — 모든 단계·분기·종착 포함.
- 단계 표: `단계 | 입력 | 처리 | 출력 | 상세 장`. 각 단계는 1문단 요약까지만 — 세부는 상세 장에 위임.
- **end-to-end 실증 예시 ≥1** — 구체 입력 1건(대표 질문·요청·레코드)이 첫 단계부터 종착까지 통과하는 과정을 산문으로 추적. 각 단계에서 그 입력이 무엇으로 변환되는지 실제 값(ID·수치·중간 산출)을 박아, 표가 설명하지 못하는 "실제로 이렇게 흐른다"를 독자가 따라가게 한다.

### 3..K_\<PART\>.md (부분별 상세 — 밀도 최대 구간)
각 장 공통 형식:
1. **위치 미니 다이어그램** — 전체 파이프라인 중 이 부분이 어디인지 강조 표시.
2. **내부 흐름 다이어그램 ≥1** — 시작→종착.
3. **구조 표** — 구성요소·개수·출처 파일 (산문 금지, 개수 박기).
4. **세부 본문** — 핵심 코드 인용(`파일경로:라인`), 설정값, 프롬프트 원문, 스키마, 엣지 케이스 처리까지. **요약으로 대체 금지** — 이 보고서는 PPT·README의 원료라 세부가 자산이다.
5. **실증 예시(walked example) ≥1** — 구체 입력 1건이 이 부분을 처음부터 끝까지 통과하는 과정을 산문으로 추적. 실제 ID·문단번호·중간 산출값을 박아 "이 질문을 넣으면 이렇게 흐른다"를 독자가 따라가게 한다(예: "리베이트 회계처리" 질문 → 용어사전 매칭 → 변동대가 개념 진입 → 관할 문단 56~58 → 상호참조 이웃 → 사례). 표·개수만으로는 전달 못 하는 동작감을 준다. **PPT·README maker가 곧바로 쓸 데모 서사가 이 절에서 나온다** — 상세 장마다 최소 1개.

### K+1_DIFFERENTIATION.md (차별점·특장점)
1. **비교 표** — `관점 | 통상적 접근 | 본 프로젝트 | 차이가 만드는 효과`.
2. **특장점 목록** — 각 항목에 근거(코드 경로·실측 수치·상술 장 링크) 필수. 근거 없는 항목은 싣지 않는다.
3. **한계·트레이드오프** — 정직하게 1섹션.

### K+2_TEST-DECISIONS.md
검증 로그 표(항목·방법·결과 **M/N**·근거 경로) + ADR 표(`결정 | 검토한 대안 | 채택 이유`).

### K+3_JOURNEY.md (아키텍처 전환 여정)
git log 기반 전환점 타임라인 — **무엇을 왜 버리고 무엇으로 대체했나**(폐기·승계). 폐기·대체된 구조는 **이 장에만, 과거형으로**. 본문 장(0~K+2)은 현재 상태만 현재형. 비-git 프로젝트는 이력 부재를 명시하고 파일에 남은 과거 흔적(legacy 디렉토리·주석·폐기 코드)만 근거로 기록한다. **운영 중 겪은 실사고·버그·성능/비용 문제는 이 장이 아니라 K+4_TROUBLESHOOTING로 보낸다**(전환 서사와 사고 서사를 분리).

### K+4_TROUBLESHOOTING.md (운영 사고·디버깅 교훈)
재구축·운영 중 **실제로 겪은 사고·버그·성능/비용 문제와 그 근본 원인·복구**를 모은다. "코드 한 줄"이 아니라 "제거·변경이 조용히 부순 것"(데이터 유실·원자성·레이트리밋·비용 폭증·성능 회귀)을 우선한다. **과거형 전용**(현재 시스템은 본문 장이 담당).
1. **사고 표** — `사고 | 증상·피해(수치) | 근본 원인 | 복구·재발 방지`. 각 행에 실측 피해 수치(유실 건수·초과 비용·지연 등)를 박는다.
2. **사고 흐름 다이어그램 ≥1** — 대표 사고 1건의 경위(원인→확산→피해→복구)를 ASCII로.
3. **교훈** — 각 사고에서 뽑은 재발 방지 원칙 1줄(예: "delete 후 재적재는 원자성이 없다").
4. **재료**: `docs/debugging.md`·트러블슈팅 로그·회고·ADR의 사고 기록·git revert/hotfix 커밋. **없으면 "해당 사고 기록 없음"으로 명시**(날조 금지 — 없는 사고를 지어내지 않는다).

### K+5_COVERAGE.md (전수 증명)
1. 헤더: 분모 N, census 실행 커맨드, inventory가 출력한 "제외 디렉토리로 걸러진 파일" 집계.
2. **파일→장 매핑 표**: `파일(백틱 필수) | kind | 반영 장 | 비고`. 경로는 프로젝트 루트 기준 상대경로·`/` 구분자. **매핑 표는 코드펜스 밖에 둔다** — verify는 인라인 백틱 경로만 파싱하고 펜스(```) 내부는 무시한다.
3. **글롭 행**: 균일 데이터 디렉토리는 `dir/**` 한 행으로 커버 가능. 단 사유(균일 데이터 등)와 대표 표본을 다룬 장을 비고에 명시. **전부 매칭 글롭 금지** — 산문에서라도 백틱으로 감싼 맨 `` `*` ``·`` `**` ``는 verify가 모든 파일을 덮는 catch-all로 읽어 N/N 증명을 무력화하고(hollow-PASS), 이제 census verify가 이를 탐지하면 FAIL(exit 2)시킨다. 별표를 예시로 언급할 땐 백틱으로 감싸지 말 것.
4. `--exclude` 를 썼다면 제외 표(디렉토리·건수·사유) 필수 — 조용한 제외 금지.
5. 목록만(정독 면제) 파일 표 + 말미에 verify PASS 출력 붙여넣기(펜스 안 — 파싱에서 무시되므로 안전).

## 시각화 규격
- **ASCII**: 코드블록 안 박스·화살표. 노드 라벨은 짧은 의미어, 해설은 다이어그램 밖 1줄.
- **Mermaid**: flowchart·sequenceDiagram 위주(GitHub 렌더 안전 범위).
- **밀도 원칙**: 전체→세세 순서. 뒤 장으로 갈수록 다이어그램·표·코드 인용이 촘촘해져야 정상.
