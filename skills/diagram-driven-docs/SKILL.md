# Diagram-Driven Docs (데이터흐름 중심 기술문서)

## 핵심 원칙

시스템 문서는 **컴포넌트 나열이 아니라 데이터의 여정**으로 설명한다. 무언가가 어디서 들어와(시작) → 어떻게 흐르고(흐름) → 어디에 닿는지(종착)를 **다이어그램으로 먼저 보이고**, 산문은 그 다이어그램을 해설한다. 모든 수치는 본문에 쓰기 전에 정본을 확정한다.

## 산출물 계약 (아래 6요소를 이 형태로 포함)

이건 금지 목록이 아니라 **산출물이 무엇인지**에 대한 레시피다. 각 요소를 빠짐없이 넣는다.

### 1. 정본 팩트시트 먼저 (facts-first)
본문 집필 전에 핵심 수치를 **출처와 함께** 한 곳에 모은다(scratchpad 파일 또는 문서 상단 표). 본문의 모든 숫자는 이 팩트시트와 일치해야 한다. 요약본·기억이 아니라 실제 파일/코드/실측에서 뽑는다. "존재≠사용" — 값이 있다고 그 값이 쓰인다는 뜻은 아니므로 사용처를 확인한다.

### 2. 데이터흐름 다이어그램 (필수, 프로세스/파이프라인마다 ≥1)
파이프라인·프로세스·시스템을 설명하는 섹션은 **시작→흐름→종착 다이어그램을 1개 이상** 반드시 포함한다. 순수 ASCII(터미널·GitHub 어디서나 렌더) 기본, 흐름도만 Mermaid도 허용. 화살표는 데이터/제어의 실제 이동을, 분기는 조건을 나타낸다.

```
  입력 ─▶ [단계 A] ─▶ 조건? ─┬─예─▶ [단계 B] ─▶ 종착(저장/응답)
   │         │              └─아니오─▶ 분기 처리
  진입점    각 단계 아래에 역할·근거 1줄
```

### 3. 데이터 여정 순서로 구성
목차를 컴포넌트 목록이 아니라 **데이터가 지나는 순서**로 짠다(수집→가공→저장→조회→표시 등).

### 4. 구조는 노드/간선·유형 표로
구성요소(노드 종류·관계·유형)는 산문이 아니라 **개수를 박은 표**로 제시한다. 예: `노드 유형 | 개수 | 출처 파일`.

### 5. 검증은 M/N으로
"모두·전수·일치" 같은 집합 주장은 분모를 박는다(예: "9/9 PASS", "차집합 0"). 검증 로그를 표로 병기한다.

### 6. 현재 vs 과거 분리
본문은 **현재 상태만** 현재형으로 서술한다. 폐기·대체된 것은 별도 "여정/변경 이력" 섹션에만 과거형으로 둔다. 폐기된 기술을 본문에서 현재 동작처럼 쓰지 않는다.

## Quick Reference

| 요소            | 넣는 위치         | 실패 신호                 |
| --------------- | ----------------- | ------------------------- |
| 팩트시트        | 집필 전 별도/상단 | 본문 수치가 출처 불명     |
| 흐름 다이어그램 | 각 프로세스 섹션  | 섹션에 다이어그램 0개     |
| 여정 순서 목차  | 문서 뼈대         | 컴포넌트 알파벳순 나열    |
| 구조 표         | 구성 설명         | 노드/유형을 산문으로만    |
| M/N 검증        | 주장 옆           | "전부 일치"인데 분모 없음 |
| 현재/과거 분리  | 별도 섹션         | 폐기 기술이 본문 현재형   |

## 흔한 실패

- **다이어그램 없이 번호 리스트로만 흐름 설명** → 시작→종착 시각화 1개 이상 추가.
- **수치를 요약 기억에서 씀** → 원본 파일/코드 파싱으로 팩트시트 먼저.
- **폐기된 구조를 "현재"로 서술** → 여정 섹션으로 이동, 본문은 현재만.
- **다이어그램에 코드/장황한 라벨** → 노드는 짧은 의미 라벨, 해설은 다이어그램 밖 1줄.

## 규모 조절

한 줄짜리 유틸 문서엔 팩트시트·다이어그램이 과할 수 있다. **데이터가 2단계 이상 흐르거나, 구성요소가 3종 이상이거나, 수치 주장이 있으면** 위 계약을 전부 적용한다. 그 미만이면 표 하나로 충분.
