# readme-maker — 인터뷰 기반 README 공장

정리본만으로는 알 수 없는, **사용자 머릿속에만 있는 "이 README에서 가장 보여주고 싶은 것"을
인터뷰로 뽑아 계약으로 박제**하고, 계약대로만 생성한다(deck-outline-grill 이식).
PPT의 차트·다이어그램 자리는 **ASCII·Mermaid·표·배지·스크린샷이 대신**하며, 시각 어휘를
다양하게 섞어야 읽는 사람 가독성이 산다.

**왜(실측 근거):** 이 게이트 없이는 에이전트가 **질문 0회로 생성에 직행**한다(2026-07-12
베이스라인 실측). 구성 변경은 생성 후가 아니라 생성 전 텍스트 단계에서 확정하는 게 가장 싸다
(pptmaker 골든 덱 동일 실측).

## ① 재료 확인 · 팩트시트

- 정리본 경로와 대상 리포지토리(README가 들어갈 곳)를 사용자에게 확인.
  **정리본이 없으면 생성을 진행하지 않고 요청한다.**
- 정리본에서 핵심 수치·기능·아키텍처를 **출처와 함께 팩트시트로** 추출. 본문의 모든 수치는
  팩트시트와 일치해야 한다.

## ② 목차·콘텐츠 인터뷰 (GRILL)

1. **목차 추천 1안 선제시** — 정리본 기반으로 스킬이 먼저 던진다(빈 종이에서 묻지 않는다).
   사용자가 수정/승인하면 구조 확정.
2. **섹션 단위 내용 질문** — "이 섹션엔 어떤 내용 위주로 구성하고 싶어?" 개방형으로 묻고
   답을 **그대로 채록**한다(요약·의역으로 왜곡 금지). 답변 원문이 그 섹션의 핵심 메시지.
3. **스크린샷 질문** — "여기에 어떤 스크린샷을 쓰면 좋을까? 파일 줄래?" 받은 파일은
   리포지토리 `docs/images/`에 복사하고 상대경로로 참조. 재료 없는 섹션은 계약에 넣지 않는다
   — "추가로 뽑을까/재료 줄래?"를 먼저 묻는다.
4. **시각 요소 제안** — 섹션마다 어울리는 유형(ASCII/Mermaid/표/배지/스크린샷/접기)을
   제안하고 확인받아 계약에 기록.
5. **한 번에 질문 하나.** 내부 용어가 보이면 "독자 언어로 바꿀까?"를 묻는다.
   **전 섹션 상태 "확정" 전에는 ③ 진입 금지.**

산출물: 대상 리포지토리 `_workspace/readme-outline.md` (커밋 제외 권장)

```markdown
# README 아웃라인 계약 — <프로젝트명>
- 대상 독자: <누가 읽는가>
- 목차(확정): 소개 → 아키텍처 → 사용법 → 성능
- 상태: 확정 (YYYY-MM-DD, 사용자 "확정" 발화)

| # | 섹션 | 내용 위주 (육성 채록 원문) | 시각 요소 | 스크린샷/재료 | 상태 |
| - | ---- | -------------------------- | --------- | ------------- | ---- |
| 2 | 아키텍처 | "수집→탐지→알림 흐름이 한눈에 보여야 해" | Mermaid 흐름도 | docs/images/pipeline.png | 확정 |
```

## ③ 초안 생성

- **REQUIRED SUB-SKILL:** diagram-driven-docs — 팩트시트 일치·흐름 다이어그램·여정 순서
  목차·구조 표·M/N 검증·현재/과거 분리 계약 전부 적용.
- **시각 선택 기준:** 간단 흐름·트리 = 순수 ASCII, 복잡 흐름도 = Mermaid(GitHub 네이티브
  렌더), 문서 상단 = shields.io 배지 스택, 결과·비교 = 표, 부가 상세 = `<details>` 접기.
- **시각 다양성 게이트 (전부 충족):**
  - 시각 유형 **최소 4종** 사용(표·ASCII·Mermaid·배지·스크린샷·접기 중)
  - 연속 두 섹션이 같은 시각 유형만 반복 금지
  - 파이프라인·아키텍처 섹션은 다이어그램 ≥1 필수
  - 산문 15줄 이상 연속(텍스트 벽) 금지 — 시각 요소나 표로 쪼갠다
- 계약 밖 섹션 추가·삭제·메시지 변경 금지. 다듬는 건 문장이지 구조가 아니다.

## ④ humanize 자동 윤문 (묻지 않고 실행)

- **보호 추출:** 코드펜스·Mermaid 블록·표·배지·HTML 블록은 윤문 대상에서 제외하고
  **산문 단락만** humanize-korean 스킬에 전달한다(분량별 모드는 그 스킬이 결정).
- 윤문 결과를 재삽입한 뒤 **diff로 코드블록·수치 불변을 확인**한다. 수치가 변했으면 롤백.
- 이 단계까지 끝나야 "README 완성"이다 — 윤문 전 초안을 완성본으로 보고하지 않는다.

## 규칙

- **채록 원문 보존** — 계약에는 육성 원문, 다듬기는 ③의 일.
- 후속 "목차 다시 / 섹션 수정 / 스크린샷 교체"는 계약 파일을 수정하고 ③부터,
  "윤문만 다시"는 ④만 재실행.
- 사용자가 인터뷰 생략을 명시한 경우에만 ②를 건너뛴다 — 그 외 "간단한 프로젝트라서",
  "정리본이 충분해서" 같은 자체 판단으로 생략 금지.

## 자체 점검

① 전 섹션 채록 채움 + 사용자 "확정" 발화? ② 시각 유형 ≥4종, 파이프라인 섹션 다이어그램 ≥1?
③ humanize 실행 완료 + 코드·수치 불변 diff 확인? ④ 계약 밖 섹션 0개? 하나라도 아니오면
해당 단계로 돌아간다.
