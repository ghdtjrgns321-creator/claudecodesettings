# readme-maker — 인터뷰 기반 README 공장

정리본만으로는 알 수 없는 **"이 README에서 가장 보여주고 싶은 것"을 인터뷰로 뽑아 계약으로
박제**하고, 계약대로만 생성한다. **게이트 없으면 에이전트는 질문 0회로 생성에 직행한다**(2026-07-12
실측) — 구성은 생성 후가 아니라 생성 전 텍스트 단계에서 확정하는 게 가장 싸다.

## ⓪ 스타일 원기(골드 예시) — 생성·수정 전 필독

`C:\Users\ghdtj\workspace\portfolio\k-ifrs-1115\README.md`가 사용자가 손으로 완성한 정답 샘플이다.
**구조·밀도·톤·깊이의 기준은 아래 규칙 문장이 아니라 이 파일** — 매 생성·수정 전에 열어 형태를
눈에 담고 시작하고, 규칙 해석이 원기와 충돌하면 원기가 이긴다. 원기의 형태(파일 못 열 때 최소 요약):

- 전체 ~400줄. 섹션 = 리드 1~2문장 + 다이어그램/표 1개 + 산문 ~50줄.
- 실증 예시 1건 — 실제 입력 원문을 파이프라인 끝까지 걸어 보인다.
- 핵심 의사결정 서사 — 버린 접근과 전환 이유(before/after)를 별도 섹션으로.
- 코드 언급은 함수·설정 키 이름 + 쉬운 뜻까지. 파일경로:줄번호(file:line) 인용 0건.
- 정직 재료(미달·한계·미재현)는 검증·한계 섹션에 버킷으로 집중 — 기술 섹션마다 "> 한계"
  콜아웃을 산개하지 않는다(원기: 미재현 14건 버킷이 검증 섹션 한 곳).

## ① 재료 확인 · 팩트시트

- 정리본 경로·대상 리포지토리를 확인. **정리본이 없으면 진행하지 않고 요청한다.**
- 정리본에서 핵심 수치·기능·아키텍처를 **출처와 함께 팩트시트로** 추출. 본문 모든 수치 = 팩트시트 일치.

## ② 목차·콘텐츠 인터뷰 (GRILL) — 한 번에 질문 하나

1. **목차 추천 1안 선제시**(빈 종이에서 묻지 않는다). **정리본에 운영 사고(데이터 유실·비용 폭증·
   성능 회귀)가 있으면** "트러블슈팅"(사고→근본원인→복구)을 기술 스택 앞에 기본 배치 — 없으면 안
   넣음(사고 창작 금지). "실패를 어떻게 다뤘나"는 강한 재료다.
2. **섹션 내용 개방형 질문** — "어떤 내용 위주로?" 답을 **그대로 채록**(왜곡 금지). **정직 재료**는 개수 말고
   과정·판단까지: 큐레이션(사전·데이터셋)=**만든 과정+사람 판단(애매 N→채택/수정/보류)** · 검증=**미달분
   버킷+근본원인+'고칠 수 있나'** · 버린 설계=**왜 바꿨나·before/after**. 불리 지표도 정의로 해명(숨김 금지).
3. **스크린샷** — 파일 받아 `docs/images/`에 복사·상대경로 참조. 재료 없는 섹션은 계약에 안 넣음.
   스크린샷·실증 예시엔 **실제 입력(질문 원문)도 함께 받아** 병기(입력 없으면 맥락이 빈다).
4. **시각 요소 제안** — 섹션마다 유형(ASCII/Mermaid/표/배지/스크린샷/코드 발췌/접기) 제안·확인해 계약 기록.
5. **스타일 계약** — 톤(담백↔서사)·구조 밀도·제목형은 **전역**, **깊이(altitude)는 섹션별**로 물어
   박제(개요=담백 골격 ↔ 심화=코드 식별자·임계까지). 정리본은 *내용*의 원천이지 톤·구조·깊이의
   원천이 아니라, 안 물으면 중간 깊이 산문이 기본값이 돼 양방향 재작업이 반복된다.
6. **눌러담기(over) 지시** — "일단 다 담아줘, 내가 선별할게"는 위 2(내용 선별)만 대체한다. 계약
   상태에 `over(선별 전 원재료)`를 기록하고 **③의 길이 예산만 해제한다** — ⓪ 문체 규칙(file:line
   0건·한 사실 한 곳·콜아웃 집중)은 over에도 그대로 적용된다.

내부 용어는 "독자 언어로 바꿀까?" 확인. **전 섹션 "확정" 전 ③ 진입 금지.** 산출물:
`_workspace/readme-outline.md`

```markdown
# README 아웃라인 계약 — <프로젝트명>
- 대상 독자 / 목차(확정) / 상태(YYYY-MM-DD 사용자 "확정" 발화)
- 스타일(확정): 톤=<담백/서사> · 구조 밀도=<세분/긴섹션> · 제목형=<기능/브랜드> (깊이는 섹션 표)

| # | 섹션 | 내용 위주 (육성 채록 원문) | 깊이 | 시각 요소 | 스크린샷/재료 | 상태 |
| - | ---- | ---- | ---- | ---- | ---- | ---- |
| 2 | 아키텍처 | "수집→탐지→알림 흐름이 한눈에" | 골격 | Mermaid | docs/images/pipeline.png | 확정 |
```

## ③ 초안 생성

**REQUIRED SUB-SKILL:** diagram-driven-docs(팩트시트 일치·M/N·현재/과거 분리). 시각 선택: 흐름=ASCII,
복잡=Mermaid, 상단=배지, 비교=표, 부가=`<details>`, **코드 주제(스키마·검증기)=주석 단 실제 발췌**(ASCII 재구성 금지·발췌/일치 명시).

- **시각 다양성** — 유형 ≥4종, 연속 두 섹션 같은 유형 반복 금지, 파이프라인·아키텍처 다이어그램 ≥1,
  산문 15줄 연속(텍스트 벽) 금지, **섹션 리드 문단 = 핵심 1개·1~2문장**(대비·나열은 다음 문단·표로).
- **사실 정합성** — ⓐ **수치 지시대상**: 숫자∈팩트시트 + 그 숫자의 주어까지 소스 라벨과 일치(비슷한 값
  여럿이면 라벨로 못박음). 숫자 맞고 주어 다르면 FAIL(오귀속). ⓑ **런타임 주장 코드 대조 = 내부
  절차, 본문 비노출**: 제어흐름 순서·런타임 개수·행동은 코드로 왕복 대조해 확인하되, 대조 기록
  (주장→file:line)은 `_workspace/readme-factcheck.md`에 남기고 **본문에는 검증된 결과 문장만 담는다**.
  독자에게 코드 위치를 알릴 땐 함수·설정 키 이름+쉬운 뜻으로 쓴다("`select_annual_report`가 정정 전
  원본 접수분을 고른다"). 본문의 file:line 인용은 원기 0건 — 검증 의무를 본문 인용으로 때우면 FAIL.
  ⓒ **재드로잉 금지** — 소스에 옳은 다이어그램 있으면 재사용(재드로잉=재유도).
- **구조·톤** — 개요는 파이프라인 N단계를 **번호 리스트로 먼저**(서사보다 골격). 나열형 결론(3항목+)은
  **불릿·표**로. **다이어그램 구성**: 복잡한 구조는 예시 1개를 끝까지 걸어본 뒤 집계로 일반화(추상 스키마
  통째 금지), 여러 출력 노드는 flat 나열 금지·목적지/역할 축으로 묶고 요소마다 "역할+어디로"·순서엔 이유.
  **심화 깊이만** 코드 식별자·임계 노출(명명+쉬운 뜻 병기, raw 코드 단독 금지; '골격'엔 금지—담백 충돌).
  **스타일 계약 준수**: 담백이면 은유·대비 수사·과장 명사 금지.
- **한 사실 한 곳** — 같은 상세(축·게이트·임계·판정 분기)는 다이어그램과 산문 중 **한 곳에만** 담는다.
  파이프라인 그림에 넣은 상세를 기술 섹션 산문이 다시 풀면 중복 FAIL — 두 곳이 필요하면 한쪽은
  1줄 요약+섹션 링크로 대신한다.
- **길이 예산** — 전체 300~500줄(원기 ±25%). 초과분은 `<details>`·별도 문서 링크로 밀어낸다.
  해제는 over 계약일 때만(문체 규칙은 유지).
- 계약 밖 섹션 추가·삭제·메시지 변경 금지 — 다듬는 건 문장이지 구조가 아니다.

## ④ humanize 자동 윤문 (묻지 않고 실행)

- **보호 추출**: 코드펜스·Mermaid·표·배지·HTML은 제외, **산문 단락만** humanize-korean에 전달.
- 재삽입 후 **diff로 코드·수치 불변 확인**(변하면 롤백). 담백 톤이면 윤문이 수사·과장 명사를 *추가*하지
  않는다(톤 유지, 부풀면 롤백). 이 단계까지 끝나야 "완성"이다.

## 규칙 · 자체 점검

- **채록 원문 보존**(계약=육성 원문, 다듬기는 ③). 후속 "목차/섹션/스크린샷 다시"=계약 수정 후 ③부터,
  **"윤문만 다시"=④만**. ②생략은 사용자 명시 때만("간단해서" 같은 자체 판단 금지).
- **후속 편집도 계약 경유** — 계약 파일이 있는 리포에서 README를 고치는 **모든** 요청(섹션 신설·
  상세 추가·다이어그램 교체·"README에 ○○ 내장" 포함)은 이 스킬로 처리한다: 계약 표를 먼저 고쳐
  사용자 확정을 받고 ③부터 재진입. 계약을 안 거친 편집이 쌓이면 원기 이탈을 아무 게이트도 못 본다
  (2026-07 실측: 계약본 331줄 → 계약 밖 3회차 편집으로 1,000줄+같은 내용 2벌).
- **드리프트 현행화** — 진입 시 본문에 계약 밖 섹션·구성이 이미 있으면 조용히 따르지 말고 보고 →
  계약 편입(사용자 확정) 또는 원기 방향 압축을 제안한다.
- 완료 전 확인: 전 섹션 채록+확정 발화 / 시각 ≥4종·파이프라인 다이어그램 ≥1 / 수치 지시대상·
  factcheck 노트 존재 / **본문 file:line 0건**(`rg ':\d+' README.md`로 파일경로 인용 검색) /
  한 사실 한 곳(그림-산문 중복 없음) / 정직 콜아웃이 검증·한계 섹션 밖에 없음 / 길이 예산(또는
  over 계약) / 스타일·섹션별 깊이·개요 골격·실입력 병기 / 계약 밖 섹션 0 / humanize diff. 아니면 복귀.
