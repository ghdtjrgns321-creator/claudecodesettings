# 08a. 완료 계약 (Completion-Contract) v2

> LLM이 "됐습니다"라고 말할 때 **실행 증거 없이는 대화를 끝낼 수 없게** 만드는 결정론적 장치.
> ← [08. 하네스 개요](08-harness.md) · 2026-07-16 v2 (v1 → [instruction-gate-design.md](instruction-gate-design.md), 레거시)

## 1. 문제 — "완료했다"는 말은 믿을 수 없다

코딩 에이전트의 고질병:

- **거짓 완료** — "전부 처리했습니다"라 하고 일부만 함 (조용한 스킵).
- **hollow-PASS** — 빈 집합·fallback·누락을 "통과"로 둔갑. 예) 표본 3개 보고 전체 40개를 "일치"라 단언.
- **존재 ≠ 사용** — "yaml에 키가 있다"를 "그 코드가 그 키를 읽는다"로 착각.

## 2. v1의 교훈 — 문장 심사는 세탁을 못 잡는다

v1은 계약 **문장**을 정규식으로 심사했다(contract_lint: 수치·실패조건·분모·보정 문구 검사). 두 가지가 실측됐다:

1. **오차단** — 일은 맞는데 문구가 규격에 안 맞아 차단 → 작문 왕복. 게이트 차단 835건(84세션)의 대부분이 이것. 1작업 1시간의 주범.
2. **미검출** — 공허한 측정(grep이 엉뚱한 데 걸려 18/19를 19/19로 보고)은 문장 형식이 완벽해서 통과. hollow-PASS의 본질이 "검증 안 된 주장을 검증된 문장으로 세탁"인데, 문장 심사는 세탁을 못 잡고 세탁 기술만 올린다.

→ v2 원칙: **검사 대상은 문장이 아니라 실체(명령의 exit 코드)다.**

## 3. 구성 (1 훅 + 1 계약파일)

```
착수 ──── 에이전트가 contract 작성   (~/.claude/state/contracts/<session>.md)
          │   done 정의 = 셀 수 있는 [ ] 항목
          │   검증 = "VERIFY: <실행 명령>" 줄
          ▼
종료 ── Stop ── completion_gate.sh v2
          │   ① 미체크 [ ] 있으면 → block
          │   ② VERIFY: 줄을 각각 직접 실행 → exit≠0 있으면 block (출력 꼬리 첨부)
          ▼
        둘 다 통과 → 종료 허용
```

계약 경로는 **HOME 고정** (`~/.claude/state/contracts/`). v1의 cwd 추종은 세션 중 `cd` 하면 게이트가 다른 곳을 보고 조용히 열리는 버그였다(2026-07-16 실측 — 계약 155건이 pptmaker/golden, plugins/cache 등에 흩어져 있던 원인).

## 4. 계약 형식

```markdown
# Contract: <작업>

- [ ] <대상> <수치> <비교> ... 실패조건: <무엇이면 FAIL>
- [ ] 2+ 케이스로 검증 (ripple-search)          ← 코딩작업이면 포함

VERIFY: python scripts/check_output.py          ← 게이트가 직접 실행 (한 줄 = 한 명령)
VERIFY: test -f data/population.txt
```

- **VERIFY는 한 줄 = 한 명령.** exit 0이어야 통과. 여러 줄 스크립트는 파일로 빼고 그 파일을 호출.
- 전수·다건 작업이면 [census 스킬](../skills/census/SKILL.md) 절차와 결합: 분모 파일 고정 → 항목 흔적 → `VERIFY: python .../census_diff.py population.txt done.txt`
- 명령으로 검증 불가능한 주장은 [x]로 세탁하지 말고 **`미검증` 라벨**을 남긴다.

## 5. 상태 문법 (체크박스)

| 표기             | 의미                 | 게이트                        |
| ---------------- | -------------------- | ----------------------------- |
| `- [ ]`          | 미완                 | **차단**                      |
| `- [x]`          | 완료                 | 통과 (VERIFY가 실체 검증)     |
| `- [>]`          | 대기(비동기 진행 중) | 통과, 같은 상태당 1회만 상기  |
| `- [~] 사유`     | 의도적 축소          | 통과 (조용한 다운스코프 방지) |
| `OVERRIDE: 사유` | 전체 포기            | 강제 통과 (로그 남음)         |

## 6. 자기조절 — 무한루프 방지 (v1에서 유지)

- **백스톱**: 같은 계약이 **N=3회 무진행**으로 재차단되면 → 로그 남기고 **자동 통과**. VERIFY 실패에도 동일 적용.
- **진행 감지**: 계약 내용이 바뀌면 카운터 리셋.
- **mute**: 백스톱·상기 후 같은 상태는 조용히 통과(재차단으로 컨텍스트 오염 방지).
- **fail-open**: 훅 오류 시 통과.

## 7. 한계

- **계약을 아예 안 쓰면 게이트가 안 걸린다** (미작성=미강제). v1은 이걸 매 턴 주입으로 압박했으나 주입 비용이 더 컸다 — v2는 CLAUDE.md의 원칙 + census 스킬 트리거에 맡긴다.
- **VERIFY 명령 자체가 공허할 수 있다** (엉뚱한 걸 세는 grep도 exit 0). 문장 심사로 막으려던 v1의 시도는 실패했다 — 대안은 명령의 품질을 리뷰(사용자·반증 에이전트)로 보는 것이지, 정규식이 아니다.

## 8. 채택법

1. `completion_gate.sh`(Stop) 하나만 배치, `settings.json` Stop 이벤트에 등록.
2. 계약 경로: `~/.claude/state/contracts/<session>.md` (HOME 고정).
3. `.gitignore`에 `state/` 추가.
