# 08a. 완료 계약 하네스 (Completion-Contract) ⭐

> 하네스의 스타. LLM이 "됐습니다"라고 말할 때 **증거 없이는 대화를 끝낼 수 없게** 만드는 결정론적 장치.
> ← [08. 하네스 개요](08-harness.md)

## 1. 문제 — "됐다"는 말은 믿을 수 없다

코딩 에이전트의 고질병:

- **거짓 완료** — "전부 처리했습니다"라 하고 일부만 함 (조용한 스킵).
- **hollow-PASS** — 빈 집합·fallback·누락을 "통과"로 둔갑. 예) 표본 3개 보고 전체 40개를 "일치"라 단언.
- **존재 ≠ 사용** — "yaml에 키가 있다"를 "그 코드가 그 키를 읽는다"로 착각.

CLAUDE.md에 "검증해라"라고 **써두는 것만으로는 부족하다.** 지시는 대략 80%만 지켜진다(업계 통설). 그래서 이 하네스는 **권고가 아니라 훅으로 강제**한다.

## 2. 핵심 아이디어 — 완료를 "계약"으로 만든다

작업 착수 전, 에이전트가 **완료 계약(contract)** 을 먼저 선언한다. 계약은 "무엇을 하면 done인지"를 **셀 수 있는 항목**으로 못박은 체크리스트다. 종료 시 게이트가 계약을 검사해서, 미달이면 **종료 자체를 차단**한다.

즉 "다 했어요" → **게이트: "계약 항목 다 [x]인가? 측정 근거 있나? 없으면 못 나감."**

## 3. 구성 요소 (3 훅 + 1 계약파일)

```
착수 ── UserPromptSubmit ─┤ ① completion_discipline.sh
                          │    매 턴 "산출물이면 계약을 [ ]로 선언" 규율 주입(soft)
                          ▼
                        ② contract 파일   (.claude/state/contracts/<session>.md)
                          │    이번 작업의 done 정의 = 셀 수 있는 [ ] 항목들
                          ▼
종료 ── Stop ─────────────┤ ③ completion_gate.sh   (강경·결정론)
                          │    계약 파싱: 미체크 [ ] 있으면 → 종료 block
                          │    모두 [x]면 → ④ 호출
                          ▼
                        ④ contract_lint.py
                             [x]인데 측정 근거 없는 "가짜 완료" 형식 검출 → block
```

## 4. 계약 형식 — 셀 수 있어야 한다

각 항목 = **대상 + 수치 + 비교/임계 + 실패조건.** "검토·확인·대조" 같은 동사형 도달 항목은 금지(측정 불가라 게이트가 차단).

```markdown
# 완료 계약: <작업>

- [ ] <대상> <수치> <비교> ... 실패조건: <무엇이면 FAIL>
- [ ] 2+ 케이스로 검증 (ripple-search)          ← 코딩작업이면 필수
```

**나쁜 예 → 좋은 예**

| ✗ 측정 불가       | ✓ 측정 가능                                            |
| ----------------- | ------------------------------------------------------ |
| "갭 표 대조 작성" | "DART 합계와 material 합계 차액 ≤ 0원, 불일치 시 FAIL" |
| "스킬 매핑 확인"  | "40행 대조표(경로 명시), 40/40 판정, 미분류 0"         |

## 5. 게이트가 막는 3가지 (contract_lint 규칙)

`[x]`로 완료 선언했어도 아래 형식이면 차단한다:

1. **측정불가 동사형** — "검토/분석/대조" 동사만 있고 수치·비교·실패조건·파일경로가 하나도 없음.
2. **분모 없는 집합 주장** — "모두·전부·매핑 일치"인데 **M/N**(확인/전체)도, 산출물 경로도 없음 → 표본→전체 점프(hollow-PASS).
3. **모집단 미등록 전수 단언** — "전수 완료"인데 **모집단(N+권위출처+증거)** 등록과 **음의공간 증명**(모집단 밖 영향원 0건)이 없음 → wrong-universe(검사-후-모집단 날조).

## 6. 상태 문법 (체크박스)

| 표기             | 의미                 | 게이트                        |
| ---------------- | -------------------- | ----------------------------- |
| `- [ ]`          | 미완                 | **차단**                      |
| `- [x]`          | 완료                 | 통과 (단 ④ 측정검사 받음)     |
| `- [>]`          | 대기(비동기 진행 중) | 통과, 같은 상태당 1회만 상기  |
| `- [~] 사유`     | 의도적 축소          | 통과 (조용한 다운스코프 방지) |
| `OVERRIDE: 사유` | 전체 포기            | 강제 통과 (로그 남음)         |

## 7. 자기조절 — 무한루프 방지

- **백스톱**: 같은 계약이 **N=3회 무진행**으로 재차단되면 → 경고 로그 남기고 **자동 통과**. 종료 보장 → 루프 불가.
- **진행 감지**: 계약 내용이 바뀌면(=한 칸이라도 체크) 카운터 리셋 → 다시 N회 기회.
- **mute**: 백스톱·상기 후 같은 상태는 조용히 통과(재차단으로 컨텍스트 오염 방지).
- **fail-open**: 훅 오류·파이썬 부재 시 통과.

## 8. 실제 예시 (이 저장소를 만들며 실제로 일어난 일)

스킬 정리 중 계약에 `- [x] 잔존 14개 전부 실물 존재` 라 썼다.

→ 게이트 **차단**: *"전수 단언인데 모집단 미등록. 누락: 모집단 등록(N+권위+증거) / 음의공간 증명."*

수정 후 통과:
```
- [x] 모집단 N=28(권위: 삭제 전 skill-rules.json 키수) = 잔존14 + 죽은14. 결과 잔존=14
- [x] 음의공간 증명: ①잔존키 실물없음=0 ②죽은14키 잔존=0 ③죽은14키 실물존재=0 → 3중 PASS
```
표본을 전체로 점프하던 습관이 게이트에서 실제로 걸렸고, 분모와 음의공간을 박고서야 통과했다.

## 9. 한계 (정직하게)

- **lazy contract는 못 막는다.** 게이트는 "선언한 걸 했나"를 보지, "옳게 선언했나"는 못 본다. → ④ 형식검사 + 매 턴 self-검수(①)로 보완하나 완전성 증명은 아니다.
- **"산출물 작업 판정"은 휴리스틱.** 계약을 아예 안 만들면 게이트가 안 걸린다(미작성=미강제).
- 설계 상세: [`instruction-gate-design.md`](instruction-gate-design.md).

## 10. 채택법

1. 훅 3개 배치: `completion_discipline.sh`(UserPromptSubmit), `completion_gate.sh`(Stop), `contract_lint.py`(gate가 호출).
2. `settings.json`에 UserPromptSubmit·Stop 이벤트로 등록.
3. 계약 경로: `{cwd}/.claude/state/contracts/<session>.md` (프로젝트별 격리, 폴백 `~/.claude/state`).
4. `.gitignore`에 `.claude/state/` 추가.
